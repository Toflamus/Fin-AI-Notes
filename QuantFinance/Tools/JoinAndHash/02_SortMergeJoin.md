# Sort-Merge Join 详解

> 系列第 2 部分。承接 [Hash Join 基础](01_HashJoin.md)，本篇剖析另一条 Join 主线：基于排序的合并。

---

## 核心思想

**如果两张表都按 join key 排好序，那么只需要各扫描一次就能完成 join。**

就像合并两个有序数组——用两个指针从头到尾各走一遍，复杂度 O(n+m)。

---

## 执行过程

以 `A JOIN B ON A.id = B.id` 为例：

### 阶段一：Sort（排序）

```
A 表排序后：          B 表排序后：
┌────┬──────┐        ┌────┬──────┐
│ id │ city │        │ id │ name │
├────┼──────┤        ├────┼──────┤
│  1 │ 东京 │        │  1 │ Bob  │
│  2 │ 北京 │        │  3 │ Eve  │
│  3 │ 上海 │        │  3 │ Amy  │  ← 重复 key
│  5 │ 广州 │        │  5 │ Zoe  │
│  7 │ 深圳 │        │  9 │ Tom  │
└────┴──────┘        └────┴──────┘
```

### 阶段二：Merge（合并扫描）

用指针 `i`（扫描 A）和 `j`（扫描 B）从头开始：

```
步骤1: A[i]=1, B[j]=1 → 匹配！输出 (东京, Bob)
       i++, j 暂留（B 可能有重复）
       → B[j]=1 没有下一个重复，j++

步骤2: A[i]=2, B[j]=3 → A < B，i++（A 追 B）

步骤3: A[i]=3, B[j]=3 → 匹配！
       B 有重复 key=3（Eve, Amy），需要输出所有组合：
       输出 (上海, Eve)
       输出 (上海, Amy)
       i++, j 回退到 key=3 的起点（Mark 机制）

步骤4: A[i]=5, B[j]=5 → 匹配！输出 (广州, Zoe)
       i++, j++

步骤5: A[i]=7, B[j]=9 → A < B，i++

步骤6: A 扫完，结束
```

---

## 重复 Key 的处理（Mark 机制）

重复值是 Sort-Merge Join 最复杂的地方：

```
A:  [3_上海]  [3_成都]
B:  [3_Eve]   [3_Amy]   [3_Zoe]

需要输出笛卡尔积：
(上海, Eve), (上海, Amy), (上海, Zoe)
(成都, Eve), (成都, Amy), (成都, Zoe)

实现方式：
1. 找到 B 中 key=3 的起始位置，记为 mark_j
2. A 中每个 key=3 的行，都让 j 从 mark_j 重新扫到结尾
3. A 的 key=3 全部处理完，才移动到下一个 key
```

最坏情况（所有 key 相同）退化为 O(n×m)，但实际数据中极少出现。

---

## 复杂度分析

```
总代价 = 排序代价 + 合并代价

排序：O(n log n + m log m)   ← 瓶颈通常在这里
合并：O(n + m)               ← 线性扫描，极快

如果数据已经有序（有索引），排序代价为 0！
→ 直接退化为纯 O(n+m) 的合并
```

## 磁盘场景：External Sort-Merge

内存放不下时，数据库用**外部排序**：

```
阶段一：生成有序归并段（Run）
─────────────────────────────────────
把表分成能放进内存的块，每块在内存中排序后写回磁盘：

表 A（100GB）→ [Run1排好序] [Run2排好序] [Run3排好序] ...
                  ↓磁盘          ↓磁盘          ↓磁盘

阶段二：多路归并
─────────────────────────────────────
用堆（Min-Heap）同时归并所有 Run：

内存缓冲区：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ Run1 头 │ │ Run2 头 │ │ Run3 头 │  → Min-Heap 取最小
└─────────┘ └─────────┘ └─────────┘
      ↓ 取出最小值，从对应 Run 补充下一行
      → 输出到结果流（同时与 B 做 Merge）
```

磁盘 I/O 总量：**3 × 数据量**（读一次→写一次→再读一次合并）

---

## 与 Hash Join 的对比

| 维度 | Sort-Merge Join | Hash Join |
|------|----------------|-----------|
| 时间复杂度 | O(n log n)（需排序） | O(n)（已在内存） |
| 数据已有序时 | **O(n+m)**，碾压 | 仍需建哈希表 |
| 内存使用 | 可预测，流式处理 | 需装下小表 |
| 支持范围 Join | ✅ `BETWEEN`、`<`、`>` | ❌ 仅等值 |
| 结果是否有序 | ✅ 自然有序 | ❌ 无序 |
| 并行化 | 较复杂 | 非常容易 |

---

## 最适合 Sort-Merge Join 的场景

```sql
-- 场景1：join key 上有索引（已经有序，排序代价=0）
SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id  -- c.id 是主键

-- 场景2：结果需要按 join key 排序（排序复用）
SELECT * FROM A JOIN B ON A.id = B.id
ORDER BY A.id  -- 反正要排序，不如用 Sort-Merge

-- 场景3：范围 Join（Hash Join 做不到）
SELECT * FROM logs l
JOIN sessions s ON l.ts BETWEEN s.start AND s.end

-- 场景4：两张表都很大，Hash Join 内存装不下小表
```

---

**一句话总结：** Sort-Merge Join 用"排好序就只需各走一遍"的思路实现线性合并，排序是唯一代价——而一旦数据本身有序，它就是最快的 Join 算法。
