# Join 与 Hash 的关系

> 系列第 1 部分。本系列将从 Hash Join 出发，逐步覆盖 Sort-Merge / Broadcast / Distributed / pandas 实现等 Join 与 Hash 的方方面面。

---

## 为什么 Join 需要 Hash？

Join 的核心问题是：**如何快速找到两张表中匹配的行？**

最朴素的方法是嵌套循环（Nested Loop Join）——对左表每一行，扫描右表所有行找匹配，复杂度是 **O(n²)**，数据量大时极慢。

Hash Join 把这个问题变成了 **O(n)**。

---

## Hash Join 的执行过程

以 `A JOIN B ON A.id = B.id` 为例：

```
阶段一：Build（构建哈希表）
─────────────────────────────
取较小的表（假设是 B），对 join key 做哈希：

B 表:
┌────┬──────┐        哈希表（内存）
│ id │ name │  →   { hash(1) → [row...],
├────┼──────┤        hash(3) → [row...],
│  1 │ Bob  │        hash(5) → [row...] }
│  3 │ Eve  │
│  5 │ Zoe  │
└────┴──────┘

阶段二：Probe（探测）
─────────────────────────────
扫描大表 A，对每行的 join key 做同样的哈希，
直接去哈希表里查找：

A 表:
┌────┬──────┐
│ id │ city │
├────┼──────┤
│  1 │ 东京 │  → hash(1) → 命中！输出匹配行
│  2 │ 北京 │  → hash(2) → 未命中，跳过
│  3 │ 上海 │  → hash(3) → 命中！输出匹配行
└────┴──────┘
```

---

## 为什么 Hash 能加速？

| 操作 | 时间复杂度 |
|------|-----------|
| 嵌套循环 Join | O(n × m) |
| Hash Join | O(n + m) |

哈希的本质是把"比较"变成"查表"——计算一次哈希值，**O(1)** 定位到候选桶，避免了逐行比较。

---

## Hash Collision（哈希冲突）怎么办？

不同的 key 可能映射到同一个桶，数据库会在桶内做二次比较（链表或线性探测）：

```
hash(3)  → 桶 #7 → [Eve, id=3]
hash(13) → 桶 #7 → [Tom, id=13]  ← 冲突！

Probe 时命中桶 #7，再逐一比较 id 是否真的相等
```

好的哈希函数让冲突极少，平均仍是 O(1)。

---

## 内存不够时：Grace Hash Join

如果小表放不进内存，数据库会用**分区（Partition）**策略：

```
1. 用同一个哈希函数把 A、B 都分成 k 个分区
   → 能 join 的行一定落在同一分区编号里

2. 每次只把一个分区的 B 加载进内存，
   对应分区的 A 来 Probe

3. 分区处理完，写回磁盘，换下一个分区
```

这样磁盘 I/O 也是线性的，整体仍是 O(n)。

---

## 什么时候不用 Hash Join？

| 场景 | 更好的选择 |
|------|-----------|
| 数据已经按 join key 排序 | **Sort-Merge Join**（避免建哈希表） |
| 右表极小（如几行的维度表） | **Nested Loop + Index** |
| join key 是范围条件（`>`、`<`） | Hash 无法用于范围，用 Sort-Merge |
| 内存极度紧张 | Sort-Merge（内存可预测） |

---

**一句话总结：** Hash Join 用空间换时间——先把一张表"索引化"到哈希表里，再用 O(1) 查找代替 O(n) 扫描，把 Join 的复杂度从平方级降到线性级。
