# Grace Hash Join 详解

> 系列第 3 部分。承接 [Hash Join 基础](01_HashJoin.md) 和 [Sort-Merge Join](02_SortMergeJoin.md)，本篇深入磁盘场景下的 Hash Join 工业实现。

---

## 为什么需要 Grace Hash Join？

普通 Hash Join 的致命弱点：**小表必须完整装入内存**。

```
普通 Hash Join 的假设：
Build 表（小表）→ 全部加载进内存哈希表 ✅
Probe 表（大表）→ 流式扫描              ✅

现实：
小表 = 50GB，内存 = 32GB → 直接失败 ❌
```

Grace Hash Join 的目标：**两张表都可以比内存大，且磁盘 I/O 最优。**

---

## 核心思想

> **能 Join 的行，join key 的哈希值一定相同。**
> 用同一个哈希函数对两张表分区，匹配的行必然落入相同编号的分区。
> 然后每次只处理一对分区，内存压力变为原来的 1/k。

---

## 三个阶段

### 阶段一：Partition（分区）

用哈希函数 `h1` 对两张表的 join key 分别做哈希，写入 k 个分区：

```
哈希函数 h1(id) % 4  →  分成 4 个分区

表 A：                        表 B：
id=1  → h1(1)%4=1 → A_p1     id=1  → B_p1   ┐
id=2  → h1(2)%4=2 → A_p2     id=3  → B_p3   │ 匹配的行
id=3  → h1(3)%4=3 → A_p3     id=5  → B_p1   │ 一定在同一
id=5  → h1(5)%4=1 → A_p1     id=7  → B_p3   │ 编号分区里
id=7  → h1(7)%4=3 → A_p3     id=9  → B_p1   ┘
id=9  → h1(9)%4=1 → A_p1

磁盘上：
A_p0  A_p1  A_p2  A_p3
B_p0  B_p1  B_p2  B_p3
```

关键保证：`A.id == B.id` 则 `h1(A.id) == h1(B.id)`，它们一定在同一分区。

### 阶段二：Build & Probe（逐对分区做 Hash Join）

每次取一对 `(A_pi, B_pi)` 加载进内存，做普通 Hash Join：

```
第1轮：加载 A_p0 + B_p0 → 内存 Hash Join → 输出结果
第2轮：加载 A_p1 + B_p1 → 内存 Hash Join → 输出结果
第3轮：加载 A_p2 + B_p2 → 内存 Hash Join → 输出结果
第4轮：加载 A_p3 + B_p3 → 内存 Hash Join → 输出结果
       ↑
       每轮只需把较小的分区建成哈希表装进内存
```

```
内存视角（第2轮）：
┌──────────────────────────────┐
│  A_p1 的哈希表               │  ← Build（用第二个哈希函数 h2）
│  { h2(1)→[id=1,东京],        │
│    h2(5)→[id=5,广州],        │
│    h2(9)→[id=9,深圳] }       │
├──────────────────────────────┤
│  B_p1 流式读入，逐行 Probe   │  ← Probe
└──────────────────────────────┘
```

注意：Build 阶段用的是**第二个哈希函数 h2**，避免与分区哈希函数冲突。

## 完整流程图

```
            ┌─────────────────────────────────────┐
            │           阶段一：Partition          │
            └─────────────────────────────────────┘

表 A ──► [h1 分区] ──► A_p0, A_p1, A_p2, A_p3  (写磁盘)
表 B ──► [h1 分区] ──► B_p0, B_p1, B_p2, B_p3  (写磁盘)

            ┌─────────────────────────────────────┐
            │        阶段二：Build & Probe         │
            └─────────────────────────────────────┘

轮次 0：  A_p0 ──► [h2 建哈希表]
          B_p0 ──► [逐行 Probe]  ──► 结果输出
                        ↓
轮次 1：  A_p1 ──► [h2 建哈希表]
          B_p1 ──► [逐行 Probe]  ──► 结果输出
                        ↓
         ... （共 k 轮）
```

## 磁盘 I/O 分析

```
阶段一（Partition）：
  读 A、B  =  1次全表扫描
  写分区   =  1次全表写入

阶段二（Build & Probe）：
  读分区   =  1次全表扫描

总 I/O = 3 × (|A| + |B|)
```

这是理论最优——数据必须至少被读写各一次，Grace Hash Join 只多了一次写。

---

## 分区数 k 如何选择？

```
目标：每个分区的较小者能装进内存

设内存大小 = M，小表大小 = S

最小分区数 k ≥ S / M

实际留出 Build 哈希表的开销（约 1.2 倍），取：

k = ceil(S / (M × 0.8))

例：小表 80GB，内存 8GB
k = ceil(80 / (8 × 0.8)) = ceil(12.5) = 13 个分区
每个分区约 6GB，可以装进内存 ✅
```

## 分区倾斜（Skew）问题

如果 join key 分布不均匀，某个分区会远大于其他分区：

```
h1(key) % 4 的分布：
p0: ████████████████████  80%  ← 热点！装不进内存
p1: ████  10%
p2: ██    5%
p3: ██    5%
```

**解决方案：递归 Grace Hash Join**

```
p0 太大，装不进内存？
→ 对 p0 单独再做一轮 Grace Hash Join（用 h3 再次分区）

A_p0 ──► [h3 再分区] ──► A_p0_0, A_p0_1, A_p0_2 ...
B_p0 ──► [h3 再分区] ──► B_p0_0, B_p0_1, B_p0_2 ...

直到每个子分区能装进内存为止
```

## 与其他 Join 算法的对比

| 维度 | Nested Loop | Hash Join | Sort-Merge | Grace Hash |
|------|------------|-----------|------------|------------|
| 时间复杂度 | O(n×m) | O(n+m) | O(n log n) | O(n+m) |
| 两表都超内存 | ✅（慢） | ❌ | ✅ | ✅ |
| 磁盘 I/O | 极高 | 低 | 3× | **3×（最优）** |
| 支持范围 Join | ✅ | ❌ | ✅ | ❌ |
| 实现复杂度 | 低 | 中 | 中 | 高 |
| 数据倾斜敏感 | 否 | 否 | 否 | **是** |

## 实际数据库中的应用

```
PostgreSQL：Hash Join 默认实现了 Grace Hash 的分区回退
            当哈希表超出 work_mem 时自动触发 batch 分区

MySQL 8.0+：Hash Join 引入时也包含了磁盘 spill 机制

Spark：     SortMergeJoin 是默认策略，但
            BroadcastHashJoin（小表广播）= 普通 Hash Join
            ShuffledHashJoin = Grace Hash Join 的分布式版本
```

---

**一句话总结：** Grace Hash Join 用"先按哈希分治、再逐对处理"的策略，把一个装不进内存的大 Join 拆成 k 个能装进内存的小 Join，以 3× 数据量的磁盘 I/O 代价实现了线性时间复杂度。
