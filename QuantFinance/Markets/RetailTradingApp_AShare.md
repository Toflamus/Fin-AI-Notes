# A 股散户交易平台（证券 APP）的订单工具生态

> **核心定位**：本文讲的是**散户交易平台（证券 APP）**提供的订单工具——条件单、网格、止损、QMT/Ptrade——这些工具由**券商托管**，与交易所级的订单类型在架构层级上完全不同。
>
> 理解这些工具是理解散户实际交易行为、以及 A 股特有的"半自动化散户"形态的关键。

## 一、三层架构：订单到底存在哪里？

理解客户端级订单，必须先搞清楚 A 股订单的**三层托管架构**：

```
┌─────────────────────────────────────┐
│  层 3：交易所（上交所/深交所）        │  撮合 & 成交
│  ─── 接收并撮合限价单、市价单         │
└─────────────────────────────────────┘
              ↑ 报单
┌─────────────────────────────────────┐
│  层 2：券商服务器（华泰/招商/银河...）│  条件判断 & 代报
│  ─── 托管条件单、网格单、止损单       │
│  ─── 当条件满足时自动向交易所报单     │
└─────────────────────────────────────┘
              ↑ 客户端指令
┌─────────────────────────────────────┐
│  层 1：用户客户端（手机 APP / PC）    │  输入 & 确认
│  ─── 普通投资者手动下单的入口         │
│  ─── 高级用户（QMT）本地跑脚本        │
└─────────────────────────────────────┘
```

**关键差异**：
- **限价单 / 市价单** → 下单瞬间就到了**层 3（交易所）**，资金冻结，进入撮合队列
- **条件单** → 只到**层 2（券商服务器）**，资金**不冻结**，静候触发
- **QMT 脚本** → 运行在**层 1（客户端本地）**，脚本判断后向层 3 报单

## 二、条件单 ≠ 限价单（最容易混淆的概念）

这是 A 股最常见的误区之一。两者在**逻辑、位置、执行方式**上完全不同。

### 2.1 存放位置不同（最核心差异）

| | 限价单 (Limit Order) | 条件单 (Conditional Order) |
|---|---|---|
| **存放位置** | 交易所单簿里排队 | 券商服务器里"待命" |
| **资金冻结** | **立即冻结** | 触发前**不冻结** |
| **交易所知不知道** | 知道 | **不知道** |

### 2.2 逻辑属性不同

- **限价单是"交易指令"**：
  - 只有价格限制
  - "只要价格是 10 元你就给我买"
  - 价格到就成交，不管市场发生了什么

- **条件单是"逻辑脚本"**：
  - "**如果 … 那么 …**"的判断
  - **如果**：满足设定的条件（时间到 14:50 / 价格跌破支撑位 / 大盘涨 2%）
  - **那么**：券商系统自动帮你提交一份委托单给交易所

### 2.3 触发后：条件单"变成"什么？

**条件满足时，条件单会自动"分身"出一个委托单报往交易所。这个报出去的委托单，往往才是真正的限价单或市价单**。

**举例**：
- 你设置条件单：**如果** 某股票价格涨到 10.5 元（触发条件），**那么** 以 10.55 元的价格挂单买入（执行动作）
- 涨到 10.5 元之前：交易所根本不知道你想买，账上的钱是活的
- 一旦到 10.5 元：券商系统**瞬间**帮你发出一份 10.55 元的限价单

### 2.4 为什么容易混淆？

因为在设置条件单时，界面通常会让你选"**触发后以什么价格报单？**"
这时候选项里会有"限价"、"市价"、"最新价"等。很多人看到"限价"两个字，就以为条件单**就是**限价单。

**正确理解**：**条件单是容器，限价单是条件触发后被容器"吐出去"的子弹**。

## 三、限价单 vs 条件单 完整对比表

| 特性 | 限价单 (Limit Order) | 条件单 (Conditional Order) |
|------|---------------------|-------------------------|
| **状态** | 已在交易所排队 | 在券商服务器"待命" |
| **资金占用** | 立即冻结 | 触发前通常不冻结 |
| **时效性** | 仅当日有效（收盘作废） | 可设置长期有效（如 30 天内） |
| **修改/撤回** | 必须在规定时间内手动撤单 | 随时修改条件，不涉及撤单费/限制 |
| **用途** | 确定价格，立即进入撮合 | 策略交易、止损止盈、定时交易 |
| **交易所视角** | 已收到委托 | 未收到委托 |
| **典型用户** | 所有人 | 略懂策略的散户 + 轻度算法用户 |

## 四、A 股券商客户端的常见订单工具

### 4.1 条件单（最常用的客户端工具）

- **主流券商都有**：华泰（涨乐财富通）、招商（招商智远）、银河、国信、东方财富等
- **触发类型**：价格 / 时间 / 涨跌幅 / 大盘指数 / 成交量
- **触发后动作**：限价买/卖、市价买/卖
- **有效期**：当日 / N 日 / 30 日

### 4.2 网格交易（Grid Trading）

- 预设一组价格区间，在区间内**循环低买高卖**
- 每次买/卖都是一次条件单的触发
- 适合震荡市，不适合趋势市

### 4.3 止损单 / 止盈单（Stop Loss / Take Profit）

- 条件单的特殊形态
- **止损单**：价格跌破阈值 → 自动卖出
- **止盈单**：价格涨到目标 → 自动卖出
- 本质：条件单 + 触发后限价卖单

### 4.4 预约单 / 隔夜委托

- 在**非交易时段**（前一晚）挂好，开盘后自动报送
- 严格讲不是条件单——它只是把用户指令**延迟投递**给交易所
- 开盘前通常会被**批量审核**，有些情况下会被自动作废（停牌、涨跌停、资金不足）

### 4.5 极速交易客户端（QMT / Ptrade）—— 准专业用户

- 券商提供的 PC 端专业交易软件
- **支持 Python 脚本** → 本地跑策略 → 直接报单
- 能实现真正的"定时撤单"、"定量下单"、"多账户联动"
- **门槛**：通常需要 **30 万 / 50 万资产以上**才能开通
- 代表：华泰 QMT、国信 Ptrade、中泰 XTP

## 五、一个重要问题：A 股散户能"定时撤单"吗？

### 结论

**普通散户的 App 基本都不支持原生"定时撤单"功能**。

### 为什么券商不做这个？

1. **合规与公平性**：监管层对自动化交易工具管控严格。大规模给散户提供"定时、定量"自动化工具容易落入**程序化交易合规地带**，尤其是开盘集合竞价这种敏感时段。
2. **风控责任**：撤单涉及资金回笼和交易指令的实时性，网络延迟或服务器波动导致定时撤单失败时**责任界定非常麻烦**。

### 散户的"平替"操作

| 平替 | 本质 | 是否真正"定时撤单"？ |
|------|------|---------------------|
| **条件单（带有效期）** | 到期自动作废未触发的单 | ❌ 这是"过期不报单"，不是把**已经报出**去的撤回 |
| **网格/止损单** | 价格触发的反向动作 | ❌ 触发条件是价格，不是时间 |
| **QMT / Ptrade** | Python 脚本本地判断 | ✅ 但有资产门槛 |
| **手动拼手速** | 9:19:59 盯盘抢撤 | ✅ 但不可靠 |

### 一种"类定时撤单"的特殊情况

- 隔夜委托 / 预约单：前一晚挂，但第二天开盘前**券商后台批量处理**
- 若遇到停牌、审核不通过，系统会在开盘前自动作废
- **感官上**像"定时撤掉"了，但本质是**投递前被拦截**，不是真正的撤单

## 六、与订单归属识别研究的联系

客户端工具的差异**本身就是散户 vs 机构的区分维度**：

| 工具 | 散户可用 | 机构/专业用户可用 | 作为归属识别的意义 |
|------|---------|----------------|-------------------|
| 手动限价单 | ✅ | ✅ | 无差别 |
| 条件单 | ✅ | ✅ | 条件单触发后的报单时间**接近整点/整分**，可识别 |
| 网格/止损 | ✅ | ✅ | 同上 |
| QMT/Ptrade | ❌（门槛 30-50w） | ✅ | **强信号**——能写 Python 脚本的大概率不是纯散户 |
| 本方最优/IOC 等快单 | ❌（普通 APP 不提供） | ✅ | 极强信号——见 [OrderTypes_AShare.md](../Basics/OrderTypes_AShare.md) |

**研究启示**：
- 盘中观察到的**整点/整分报单**，大概率是条件单触发——可能是**半自动化散户或轻度算法用户**
- 观察到 **U 类（本方最优）/ S 类（IOC）订单**，大概率是机构或 QMT 用户
- 这是 `Fin-Research/OrderAttribution_AShare_2026Q2/` 主题中的**行为维度 A 的重要细化**

## 七、回到层级视角的总结

| 订单类型 | 托管层级 | 资金冻结 | 典型用户 |
|---------|---------|---------|---------|
| 市价单 / 普通限价单 | 交易所 | 立即 | 所有人 |
| 本方最优 / IOC / FOK | 交易所（特殊市价申报） | 立即 | 机构 / 执行算法 |
| 条件单 / 网格 / 止损 | **券商服务器** | **触发前不冻结** | 略懂策略的散户 |
| QMT / Ptrade 脚本 | **客户端本地** | 报单时冻结 | 准专业用户（资产门槛） |
| 冰山 / 暗池订单 | 交易所（但量隐藏） | 立即 | 机构 |

**一句话**：**限价单是"已发出的信"，条件单是"放在抽屉里的便签"，QMT 脚本是"你自己写的自动回信程序"**。
